# TODO

面向生产部署与公网暴露的安全改进清单。

## P0 - 公网前必须完成

- [x] 关闭 PostgreSQL 对公网暴露
  - 调整 `docker-compose.prod.yml`，移除 PostgreSQL 的宿主机端口映射。
  - 只允许 Compose 内部服务通过 Docker 网络访问数据库。
  - 验收标准：公网只能访问反向代理暴露的 443 端口，不能直接连到 PostgreSQL。

- [ ] 通过 Cloudflare Tunnel 暴露站点并强制 HTTPS
  - 以 Cloudflare Tunnel 作为公网入口，由 Cloudflare 边缘负责 TLS 终止和 HTTP 跳转 HTTPS。
  - 源站不再自建对公网开放的 Caddy / Nginx，也不再直接暴露 `app` 宿主机端口。
  - `app` 只保留给 Docker 内部网络或本机回环访问，外部流量统一经 Cloudflare 域名入口进入。
  - 验收标准：浏览器只通过 HTTPS 访问站点；公网无法直接访问源站应用端口，只能通过 Cloudflare 域名入口访问。

- [ ] 将会话 Cookie 和登录 CSRF Cookie 改为生产安全模式
  - 为 Cookie 增加 `Secure`。
  - 显式设置 `SameSite`，优先考虑 `Lax` 或按实际需求更严格配置。
  - 保持 `HttpOnly`，并确保仅在 HTTPS 下发放敏感 Cookie。
  - 验收标准：生产环境响应中的会话相关 Cookie 带有 `Secure`、`HttpOnly`、`SameSite`。

- [ ] 补充反向代理场景下的可信代理配置
  - 在 Gin 中显式配置 `SetTrustedProxies`。
  - 审核 `ClientIP()` 的使用，确保在 Cloudflare Tunnel 转发链路下拿到真实客户端 IP，避免被伪造的 `X-Forwarded-For` 干扰登录/上传限流。
  - 验收标准：经过代理转发时，服务端拿到的客户端 IP 正确且不可被任意伪造。

- [ ] 强化登录防爆破与上传限流策略
  - 保留应用层限流，但把第一层限流前移到 Cloudflare 边缘规则或其他公网入口层。
  - 重新评估当前登录限流参数是否足够严格。
  - 视需要增加失败登录次数统计、短时封禁或额外告警。
  - 验收标准：高频登录爆破请求会在公网入口层和应用层都被明显抑制。

- [ ] 为生产启动增加弱配置拦截
  - 启动时拒绝明显占位值，如 `change-me`、默认管理员密码、过短 `SESSION_SECRET`。
  - 部署脚本也要在执行前校验 `.env.production` 的关键变量。
  - 验收标准：带默认口令或弱密钥的配置无法成功 deploy。

## P1 - 建议尽快完成

- [ ] 将无状态签名 Cookie 会话改成可撤销会话
  - 当前登出只删除浏览器端 Cookie，已泄露的会话在过期前仍可继续使用。
  - 可选方案：数据库会话表、Redis 会话、或基于服务端版本号的会话失效机制。
  - 验收标准：用户登出、改密或管理员主动失效后，旧会话立即不可用。

- [ ] 补齐 HTTP 服务超时与连接限制
  - 增加 `ReadTimeout`、`WriteTimeout`、`IdleTimeout`、`MaxHeaderBytes`。
  - 评估大文件上传场景，避免超时设置过短影响正常使用。
  - 验收标准：服务对慢连接和异常长请求的抵抗能力提升，同时不误伤正常上传。

- [ ] 降低运行容器权限
  - 生产镜像改为非 root 用户运行。
  - 收紧容器写入目录，只保留应用必需的写权限。
  - 评估 `read_only` 根文件系统、`no-new-privileges` 等运行时限制。
  - 验收标准：即使媒体处理组件出现漏洞，也只能拿到受限容器权限。

- [ ] 加入安全响应头的生产化配置
  - 在保留现有 CSP 的基础上，增加 HSTS。
  - 审查 `X-Frame-Options` / `frame-ancestors` 是否与当前 PDF 查看行为一致。
  - 验收标准：安全响应头与实际页面/文件流行为一致，不产生误拦截或明显缺口。

- [ ] 明确日志与审计策略
  - 记录登录失败、登出、上传、删除、恢复、彻底删除等关键安全事件。
  - 避免日志中泄露密码、Cookie、CSRF token、数据库连接串等敏感信息。
  - 验收标准：出现异常访问或误操作时，可以从日志快速追溯关键行为。

## 实施顺序建议

1. 先改生产部署拓扑：Cloudflare Tunnel、HTTPS、关闭数据库公网端口与源站应用公网端口。
2. 再改鉴权与启动安全：Cookie 属性、可信代理、弱配置拦截。
3. 然后补服务抗压能力：限流、超时、容器降权。
4. 最后补长期治理项：可撤销会话、日志审计、自动化测试与巡检。
